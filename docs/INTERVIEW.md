# DOTMappers Support Tickets AI — Interview Guide

## Project Summary
- What: A prototype AI system that ingests a customer support ticket dataset (CSV), makes it queryable, answers natural-language questions, detects anomalies, and exposes functionality via a REST API and a minimal web UI.
- Goal: Demonstrate end-to-end design — data ingestion → Natural Language Understanding (NLU) → execution → results / Application Programming Interface (API) / User Interface (UI) — with tests and graceful fallbacks.

## Architecture & Flow
1. Data Layer
   - The CSV (`extracted/support_tickets.csv`) is loaded into a Pandas DataFrame. Dates are parsed and numeric fields cleaned for analysis.
2. NLU Layer (Natural Language Understanding)
   - Natural Language Understanding (NLU) is the subsystem that interprets user-provided natural language questions and maps them to structured operations the system can execute.
   - When available, the Hugging Face Inference API is used to convert user questions into a structured JSON specification describing the desired operation (action, filter, group_by, agg).
   - If the Hugging Face model is not available, a deterministic rule-based fallback handles common intents (counts, groupby/aggregation, list, raw preview).
3. Execution Layer
   - The JSON spec is executed against the DataFrame using Pandas (`.query()`, `.groupby()`, `.agg()`).
   - Actions include: filter_count, groupby_agg, list, and raw preview.
4. Anomaly Detection
   - Rule-based checks: (a) resolution_time_hrs > mean + 3*std, (b) unresolved high-priority tickets older than 24 hours.
   - Results are sanitized for JSON (convert datetimes to ISO strings; convert NaN/Inf to null).
5. API & UI
      - The backend is implemented with FastAPI and exposes `POST /query` and `GET /anomalies` endpoints.
      - A static `index.html` provides a minimal front-end user interface to submit questions and view anomalies.

## Key Files & Their Importance
- `app.py` — Core backend. Loads data, defines API routes, handles Hugging Face integration, parsing, execution logic, sanitization, and CORS. Primary file to show in interview.
- `index.html` — Minimal, modern front-end used to demo the system end-to-end; demonstrates UX for asking questions and viewing anomalies.

## Design Choices & Trade-offs
- Using Hugging Face enables flexible NLU quickly but introduces external dependency and latency; fallback parser ensures reliability.
- Pandas is great for small-to-medium datasets and rapid prototyping; for scale, migrate to a DB or vector store.
- Rule-based anomalies are transparent and easy to explain; can be improved with ML approaches later.

## Potential Improvements
- Add authentication and API keys.
- Persist data in a DB (Postgres) with indexing for scale.
- Improve parsing with RAG or retrieval for contextual answers.
- Add more unit tests and integration tests, including mocking the HF API.
- Add observability (metrics, logs) and production-grade secrets handling.

## Glossary (Abbreviations Expanded)
- NLU — Natural Language Understanding: interpreting user language and mapping it to structured operations.
- LLM — Large Language Model: a deep learning model (e.g., T5, GPT) used for language tasks.
- HF — Hugging Face: platform and API provider for machine learning models.
- RAG — Retrieval-Augmented Generation: combining retrieval of documents with generation from an LLM.
- API — Application Programming Interface: endpoints/services exposed by the backend.
- UI — User Interface: the front-end interface users interact with (e.g., `index.html`).
- CSV — Comma-Separated Values: tabular data file format used for the dataset.
- DB — Database: persistent storage useful for scaling beyond in-memory DataFrames.

## Detailed Step-by-Step Implementation and Data Flow
This section lists every major step and tool used to build the prototype, and explains, for each step, what is happening, what the input is, what the output is, and which files or libraries are involved.

1) Project setup and environment
   - Tools / libraries: Python 3.10, `python -m venv` (virtual environment), `pip` and `requirements.txt`.
   - What happens: create an isolated environment and install dependencies declared in `requirements.txt` (FastAPI, Uvicorn, Pandas, Requests, python-dotenv, PyPDF2, openpyxl, pytest, httpx).
   - Input: nothing user-provided beyond the repository files.
   - Output: a reproducible runtime with required packages installed in `.venv`.

2) Data extraction (one-time step executed earlier)
   - Tools / libraries: `PyPDF2`, `pandas`, `openpyxl`.
   - What happens: extract text from the PDF instructions and convert the Excel sheet `support_tickets (2).xlsx` into a canonical CSV stored at `extracted/support_tickets.csv`.
   - Input: `AI_Engineer_Assessment_Sprint (1).docx - Copy.pdf` and `support_tickets (2).xlsx` in the project root.
   - Output: `extracted/support_tickets.csv` (tabular data read by the application) and an extracted plain-text file for the PDF if needed.

3) Data loading and preprocessing (runtime in `app.py`)
   - Tools / libraries: `pandas`.
   - What happens: when the server starts or endpoints are called, `app.py` uses `pd.read_csv("extracted/support_tickets.csv")` to load the dataset into memory as a `pandas.DataFrame`. The `created_at` column is parsed into Python `datetime` objects with `pd.to_datetime(..., errors='coerce')`. Numeric fields with missing or invalid values remain `NaN` in the DataFrame.
   - Input: `extracted/support_tickets.csv` file on disk.
   - Output: an in-memory DataFrame used by all subsequent operations.

4) Secret configuration and model selection
   - Tools / libraries: `python-dotenv` (`dotenv.load_dotenv()`), environment variables.
   - What happens: `app.py` calls `load_dotenv()` which loads variables defined in `.env` into the process environment. `HF_API_TOKEN` and `HF_MODEL` determine whether the application will call the Hugging Face Inference API and which model to use.
   - Input: `.env` file (contains `HF_API_TOKEN` and `HF_MODEL`) or environment variables configured in the shell/CI.
   - Output: token and model values accessible via `os.getenv()` in `app.py`.

5) Natural Language Understanding (question parsing)
   - Tools / libraries: `requests` to call Hugging Face Inference API (if token provided), otherwise deterministic rule-based parser implemented in `app.py`.
   - What happens: upon a `POST /query` request, the server builds a prompt that asks the model to return a JSON specification describing the action (for example, `{"action":"filter_count","filter":"priority == 'High' and status != 'Resolved'"}`) and posts it to the model endpoint. If a model response is returned, the code extracts the JSON and uses it as the execution spec. If no token is present or model fails, fallback parsing uses simple heuristics (keyword detection) to produce the spec.
   - Input: user natural language question from the UI or API request body (JSON: `{"question": "..."}`).
   - Output: structured JSON `spec` object with keys like `action`, `filter`, `group_by`, and `agg`.

6) Execution engine (apply spec to DataFrame)
   - Tools / libraries: `pandas` DataFrame operations (`.query()`, `.groupby()`, `.agg()`).
   - What happens: the `spec` is interpreted: if `action == 'filter_count'`, the server runs `df.query(filter)` and returns the row count; if `action == 'groupby_agg'`, it performs grouping and aggregation; `list` returns the first N rows; unknown or `raw` returns a preview.
   - Input: in-memory DataFrame and the `spec` produced by NLU or fallback parser.
   - Output: Python objects (integers, `pandas.DataFrame` or lists of dicts) representing the query results.

7) Sanitization and JSON preparation
   - Tools / libraries: standard Python `json` via FastAPI, `numpy`/`pandas` helpers.
   - What happens: before returning the response, numeric and missing values (`NaN`, `Inf`) are converted to `null` by replacing them with `None`. Date/time objects are converted to ISO 8601 strings. A recursive sanitizer function ensures nested structures are JSON-safe to prevent `json.dumps` errors.
   - Input: Python-native results from the execution engine.
   - Output: JSON-serializable dicts/lists returned by FastAPI as HTTP responses.

8) Anomaly detection (`GET /anomalies`)
   - Tools / libraries: `pandas`, `numpy`.
   - What happens: the endpoint computes simple, explainable anomalies:
     - Slow resolution: compute `mean` and `standard deviation` of `resolution_time_hrs`, flag rows above `mean + 3 * std`.
     - Old unresolved high-priority: find tickets where `status != 'Resolved'` and `priority` in `["High","Critical"]` and age is greater than 24 hours.
   - Input: in-memory DataFrame.
   - Output: JSON list(s) of flagged ticket records (sanitized).

9) API layer and server
   - Tools / libraries: `FastAPI`, `Uvicorn` (ASGI server).
   - What happens: `app.py` registers routes (`/` serves `index.html`, `/query` accepts POST, `/anomalies` accepts GET). `uvicorn app:app --reload` runs the server. FastAPI handles request parsing and response serialization, and middleware like CORS is configured for the UI.
   - Input: HTTP requests from the browser or curl/postman.
   - Output: HTTP responses containing JSON results or static HTML.

10) Front-end UI
    - Tools / libraries: plain HTML, CSS, JavaScript in `index.html`.
    - What happens: user types a question and clicks Ask. The JavaScript code issues a `fetch` POST to `http://127.0.0.1:8000/query` with `{"question":"..."}`. It renders the returned JSON into a user-friendly display. The Anomalies tab issues a GET to `/anomalies` and renders flagged records.
    - Input: user-typed question or button clicks in the browser.
    - Output: visual results displayed on the page (counts, tables, or anomaly lists).

11) Testing and verification
    - Tools / libraries: `pytest`, `httpx` (via FastAPI test client).
    - What happens: `tests/test_api.py` uses FastAPI's `TestClient` to call the endpoints programmatically and asserts correct HTTP status codes, presence of expected keys in responses, and simple performance thresholds.
    - Input: the running `app` module (imported directly by tests) and test code.
    - Output: test pass/fail results printed by `pytest`.

12) Security and secrets handling
    - What happens: secrets (the Hugging Face API token) are not committed. They are stored in `.env` for local development and accessed through environment variables via `os.getenv('HF_API_TOKEN')`. In production, use a secrets manager or CI secrets.
    - Input: `.env` or environment-specific secret storage.
    - Output: secure retrieval of tokens at runtime.

Flow Summary (Input → Processing → Output)
- User types a question in the browser (`index.html`) → JavaScript sends `POST /query` with JSON body → FastAPI receives request and calls parse_question() → `parse_question()` uses HF API or fallback to build `spec` → execution engine applies `spec` to in-memory DataFrame → results are sanitized → FastAPI returns JSON → browser renders result.
- For anomalies: user clicks Anomalies → JavaScript sends `GET /anomalies` → server computes rule-based anomalies → sanitizes and returns JSON → browser displays flagged tickets.
